# Contract: CLI additions (`generate`)

Additive to the existing `generate` contract (specs/001). Only the deltas are listed.

## New options

| Option | Arg | Description |
|--------|-----|-------------|
| `--auth-config` | `<file>` | Path to an `auth.json` (see [auth-config.md](./auth-config.md)). Parsed + validated, then copied verbatim into the skill dir. Mutually exclusive with `--auth`. |
| `--auth` | `<type>` | Shorthand: scaffold a single **global** profile of `bearer` \| `basic` \| `custom` into a new `auth.json` (+ `secrets.example.json` keys). `oauth2`/`entra` here ⇒ **usage error** directing to `--auth-config`. Mutually exclusive with `--auth-config`. |
| `--login` | (flag) | After a successful write, run the interactive login once for each `authorization_code` profile to prime `.auth-cache.json`. Without it, generation is fully non-interactive (CI-safe). |

`--out`/`-o` unchanged (existing explicit output-folder selection — FR-024).

## Exit codes

Existing: `0` success, `1` parse failure, `2` usage error, `3` output exists, `4` acquisition
failure. **New:**

| Code | Name | When |
|------|------|------|
| `5` | `AuthConfigError` | `auth.json` malformed, unknown profile `type`, missing required oauth2 field, or a header collision (FR-021a). No output written. |

Missing `--auth-config` file ⇒ `4` (acquisition failure) with a clear "auth config not found:
<path>" message. `--auth oauth2`/`--auth entra` ⇒ `2` (usage error).

## `--force` interaction (FR-002 / Constitution IV)

On regeneration over an existing skill dir:
- `secrets.json` — **preserved** byte-for-byte (unchanged behavior).
- `.auth-cache.json` (+ `.lock`) — **preserved** (holds live sessions).
- `auth.json` — **preserved** if the run supplies **no** `--auth-config`/`--auth`; **replaced**
  when a new one is supplied this run.

All preservation happens via the existing staging-then-move flow (no partial output on failure).

## Determinism (NFR-4)

Given the same (spec, `auth.json`, options), output is **byte-stable**: `auth.json` is copied
verbatim; per-operation profile-name lists are emitted in a deterministic order. PKCE/`state`/
tokens are runtime-only and never affect generated bytes.
