# Feature Specification: Skill Update Command

**Feature Branch**: `feature/003-skill-update-command`

**Created**: 2026-07-11

**Status**: Draft

**Input**: User description: "Once a skill has been generated with a custom name and output path, let the user update it in place from a newer version of the source OpenAPI document without having to remember and retype every original generation option."

## Overview

`api2skill generate` already supports a custom name (`--name`) and output path (`--out`), and `--force` already regenerates an existing skill directory while preserving `secrets.json`, `auth.json`, and `.auth-cache.json`. What's missing is a way to point at a **previously generated skill** and refresh it from a **newer version of its source spec** without re-supplying every flag (script kind, filters, base URL, insecure) used the first time.

This feature adds a small, secret-free **generation manifest** (`.api2skill.json`) written into every generated skill directory, and a new **`update`** command that reads it back and re-runs generation in place.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Update a skill from a new spec without retyping options (Priority: P1)

A user generated a skill weeks ago with several non-default flags (`--script fsx`, `--include tag:pet`, `--base-url https://staging.example.com`). The API's OpenAPI document has since changed (new operations, changed schemas). They want to refresh the skill from the new document without hunting down or re-typing the original flags.

**Why this priority**: This is the entire value of the feature — the whole point is *not* re-typing options.

**Independent Test**: Generate a skill with several non-default options, then run `api2skill update <skill-path> <new-spec>` and confirm the regenerated skill reflects the new spec while still honoring the original script kind, filters, base URL, and insecure setting — none of which were passed to `update`.

**Acceptance Scenarios**:

1. **Given** a skill generated with `--script fsx --include tag:pet`, **When** the user runs `api2skill update <path> <new-spec-path>`, **Then** the regenerated skill is an `.fsx` dispatcher containing only `pet`-tagged operations from the new spec.
2. **Given** the same skill, **When** `update` completes, **Then** `secrets.json`, `auth.json`, and `.auth-cache.json` (if present) are preserved exactly as `--force` already preserves them today.
3. **Given** a skill generated from a URL spec source, **When** the user runs `update <path>` with **no** new spec source, **Then** the tool re-fetches the *same* URL recorded at generation time (picking up any changes published there since).

---

### User Story 2 - Clear failure when the target isn't an api2skill skill (Priority: P2)

A user runs `update` against a directory that was never generated by api2skill (or was generated by a version predating the manifest).

**Why this priority**: A safe, clear failure here prevents `update` from doing something confusing to an unrelated directory.

**Independent Test**: Run `update` against an empty directory and against a directory containing an unrelated `SKILL.md`; both fail with a clear message and change nothing.

**Acceptance Scenarios**:

1. **Given** a path with no `.api2skill.json`, **When** the user runs `update <path>`, **Then** the command fails with a message naming the missing manifest and suggesting `generate`, and no files are written or modified.

---

### Edge Cases

- **New spec source given**: `update <path> <new-spec>` uses the given source and **records** it as the new "original" source for the *next* update (the manifest always reflects the most recent invocation).
- **Manifest present but corrupted/unreadable JSON**: treated the same as "no manifest" (User Story 2) — clear failure, no partial output.
- **`update` on a skill whose original `--auth`/`--auth-config` is no longer relevant**: out of scope — `update` never changes auth configuration; the existing `auth.json` in the target directory is preserved exactly as `--force` already preserves it. Changing auth requires `generate --force --auth-config ...` directly.
- **Renaming a skill (changing its `--name`) during an update**: explicitly out of scope for this feature (see Assumptions) — tracked separately.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Every successful `generate` MUST write a secret-free generation manifest (`.api2skill.json`) into the generated skill directory, recording at least: the resolved skill name, the spec source, the script kind, include/exclude filters, the forced input format (if any), the base URL override (if any), and the insecure-default flag.
- **FR-002**: The manifest MUST contain no real credentials or resolved secret values — it is safe to commit alongside `SKILL.md`.
- **FR-003**: A new `update <skill-path> [<spec-source>]` command MUST read the manifest from `<skill-path>/.api2skill.json` and regenerate the skill using its recorded options, without requiring the user to re-supply them.
- **FR-004**: `update` MUST accept an optional new spec source; when omitted, it MUST re-resolve the spec source recorded in the manifest (re-fetching a URL, re-reading a file) rather than requiring one.
- **FR-005**: `update` MUST regenerate using the same safety guarantees as `generate --force`: staged-then-atomic-move (no partial output on failure), and preservation of `secrets.json`, `auth.json`, and `.auth-cache.json`.
- **FR-006**: `update` MUST fail clearly, changing no files, when the target path has no readable manifest (never generated by api2skill, or predates this feature).
- **FR-007**: After a successful `update`, the manifest MUST be rewritten to reflect the invocation just performed (e.g., a newly supplied spec source becomes the recorded source for next time).
- **FR-008**: `update` MUST NOT alter auth configuration (`auth.json`) — it always regenerates with no new auth config supplied, relying on existing preservation behavior.

### Key Entities

- **Generation manifest**: the secret-free `.api2skill.json` record of the options used to produce a given generated skill, enabling later unattended regeneration.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A user can refresh a previously generated skill from a new spec version by supplying only the skill's path (and, optionally, a new spec source) — no other original flags need to be retyped.
- **SC-002**: Every artifact `--force` already preserves today (`secrets.json`, `auth.json`, `.auth-cache.json`) continues to be preserved via `update`.
- **SC-003**: Running `update` against a non-api2skill directory fails clearly and modifies nothing.
- **SC-004**: No committed manifest ever contains a real secret value.

## Assumptions

- Changing a skill's custom name or output location during an update (i.e., "renaming" it) is a distinct capability, tracked as separate future work — not built here. `update` always regenerates at the same path under the same recorded name.
- `update` never introduces or changes auth configuration; that remains `generate --force --auth-config ...`'s responsibility.
- The manifest is a new, additive file; skills generated by earlier versions of api2skill (with no manifest) require one `generate --force` before `update` can be used on them.
